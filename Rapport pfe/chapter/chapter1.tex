\subsubsection{Criticism of the existing}
Based on the comparison of the existing systems, we note the following limitations:
\begin{itemize}
    \item \textbf{Odoo}: Although Odoo offers a wide range of features, it can be complex to learn and use. In addition, creating highly customized client portals or specific mobile functionalities often requires advanced customization or additional development efforts.
    \item \textbf{ERPNext}: While it simplifies document attachments and has a solid database schema, it lacks interactive real-time messaging workspaces where managers and client companies can communicate immediately.
    \item \textbf{Dolibarr}: Dolibarr is lightweight and free, but its interface is outdated and relies on legacy web technologies. It does not provide reactive, real-time feedback for document processing states or instant support messaging.
\end{itemize}

\subsection{Proposed solution}
To address the limitations identified above—specifically Odoo's high complexity and lack of tailorable mobile client interfaces, ERPNext's absence of real-time messaging, and Dolibarr's outdated interface with no instant feedback on document states—we designed and developed a tailored application ecosystem called \textbf{HMI Stars}. The system is architected to solve these specific challenges through three integrated components:
\begin{enumerate}
    \item \textbf{A Web Application Platform:} Resolves the issues of complex or outdated interfaces (as seen in Odoo and Dolibarr) by providing HMI Stars administrators with a clean, modern, and reactive web dashboard. It streamlines company registration, organizes client files and enables immediate response to client inquiries.
    \item \textbf{A Cross-Platform Mobile Application:} Addresses the lack of intuitive mobile access for client companies. It provides client managers with a simplified, mobile-first tool to manage employee registries, record absences, track daily attendance, and upload documents directly from their mobile devices.
    \item \textbf{A Real-Time Backend Server:} Directly solves the lack of real-time synchronization and instant feedback identified in both ERPNext and Dolibarr. Powered by Supabase \cite{SupabaseDocs}, it manages instant messaging and file sharing via WebSocket streams, while ensuring absolute data isolation between client companies using PostgreSQL \cite{PostgresDocs} Row Level Security (RLS) policies.
\end{enumerate}

\subsubsection{Technology choices}
The technical track of the project required selecting a cross-platform frontend framework and a backend-as-a-service platform. 

For the frontend framework, we compared \textbf{Flutter} \cite{FlutterDocs} (Dart) and \textbf{React Native} (JavaScript). While both are open-source and support cross-platform development, Flutter enables compiling natively for Android, iOS, and Web from a single Dart codebase. Conversely, React Native often requires separate configuration and extra tooling (such as React Native Web) to ensure compatibility on web platforms, which increases development complexity for our dual-interface system.

For the backend platform, we compared \textbf{Supabase} (PostgreSQL) and \textbf{Firebase} (NoSQL). The comparison focused on three key aspects:
\begin{itemize}
    \item \textbf{Data model}: Supabase uses relational PostgreSQL tables with SQL foreign key constraints and joins, which naturally fits our structured business data. Firebase \cite{FirebaseDocs} uses a NoSQL document-based structure that is less suited for relational mappings.
    \item \textbf{Security}: Supabase offers native PostgreSQL Row Level Security (RLS) policies to enforce data isolation at the database level. Firebase relies on Firestore rules which are evaluated at the API boundary.
    \item \textbf{Real-time capability and open-source nature}: Both offer real-time synchronization, but Supabase is fully open-source and self-hostable, whereas Firebase is proprietary.
\end{itemize}

We selected \textbf{Flutter} because it enables the development of both mobile and web applications from a single codebase, reducing development time and ensuring consistency across platforms. We chose \textbf{Supabase} because its PostgreSQL database is well suited for managing structured data such as companies, employees, contracts, and attendance records. In addition, its built-in security features help ensure that each company can only access its own data.

\section{Methodological choices and work organization}

\subsection{The unified process (2TUP)}

We chose the \textbf{2TUP (Two Track Unified Process)} methodology \cite{Rocher2003} for the development of this project. This approach separates the development process into two parallel tracks: a \textbf{functional track}, focused on capturing business requirements, modeling use cases, and specifying system behavior; and a \textbf{technical track}, focused on selecting technologies, defining the architecture, and designing the database and deployment infrastructure. The two tracks converge during the Construction phase, where the functional specifications and technical decisions are merged into a working implementation.

Table \ref{tab:2tup_tracks} shows how the two tracks are distributed across the report chapters and sections.
